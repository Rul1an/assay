# assay-canonical

Deterministic canonicalization for Assay semantic digests, in composable layers:

1. **`parse_strict`** — parse raw JSON into a `serde_json::Value` while **rejecting duplicate object
   keys** at any depth. `serde_json::from_str::<Value>` silently collapses duplicates (last wins),
   erasing an ambiguity before canonicalization can reject it; RFC 8785 / I-JSON treat duplicate keys
   as invalid. This is the fail-closed entry point for untrusted JSON at the canonical boundary.
2. **`jcs`** — RFC 8785 (JSON Canonicalization Scheme) bytes via the pinned `serde_jcs`. Object keys
   are sorted; arrays are left exactly as emitted. Byte-for-byte the canonical form `assay-evidence`
   already uses for its content and mandate IDs (same `serde_jcs` pin).
3. **`content_id`** — `"sha256:" + hex(sha256(jcs_bytes))`, the content-addressed id used across
   Assay evidence. Hashes the value *as given*; arrays are not reordered.
4. **`set_paths::normalize_sets`** — schema-aware normalization: sort + dedupe ONLY the array fields a
   schema registers as semantic *sets*, before canonicalization. Returns a typed `SetPathError`
   (`EmptyPath`, `MalformedSetValue { path }`) so a fail-closed reject is auditable.
5. **`semantic_digest`** — the product-facing digest: normalize the registered set-paths, bind the
   `PROFILE` into the preimage, then content-address.

## Two digests, on purpose

- `content_id` alone is an **as-received** digest: "are these the same bytes I saw?". Callers doing
  forensic correlation use it directly and must **not** set-normalize — sorting a set-valued array
  would make genuinely different received payloads collide.
- `semantic_digest` is a **semantic-equivalence** digest under a named profile: two records that
  differ only in the order or duplication of a registered set, under the same profile, collapse to
  one id.

## Profile binding

`semantic_digest(record, set_paths, profile)` binds the `canonicalization_profile` string into the
digest preimage (v1 shape: `{canonicalization_profile, record}`, JCS-ordered). Consequences:

- a digest under one profile never collides with one under another;
- introducing or bumping the profile moves the digest with no reorder — the contract's "old/new
  golden pair" on adoption — instead of silently changing meaning;
- `profile` is a required argument (never defaulted), and a consumer calls
  `ensure_supported_profile(profile)` to fail closed (`Error::UnknownProfile`) on a profile this
  build does not implement, rather than recompute under the current rules.

`PROFILE` is `assay.semantic-digest.jcs-rfc8785.v1`, matching the contract registry.

## Conformance + Rust/Python parity

`tests/fixtures/goldens.json` is generated by the Python reference in `tests/reference/`
(`canonical.py` + `build_goldens.py`). Its digests are the **un-profiled** `content_id` of the
set-normalized record — the contract's current, pre-profile-adoption digest. `tests/conformance.rs`
reproduces every one of them from the value bytes alone and checks the contract (set classes collapse
under reorder, ordered classes do not, both directions exercised).

Reproducing the Python-generated goldens in Rust is the parity proof **over the goldens' value
domain**: ASCII strings and integers. On exactly that domain `serde_jcs` (full RFC 8785) and the
Python reference (`json.dumps(sort_keys=True, separators=(",", ":"))`) are byte-identical. Floats and
non-ASCII are out of this fixture domain — `serde_jcs` handles them per RFC 8785, but extending the
parity oracle to them would mean upgrading the Python reference to a full RFC 8785 implementation.
That is a follow-up. The profile-bound `semantic_digest` is exercised in the crate's `profile` unit
tests, not in the goldens, because the goldens pin the pre-profile bytes.

To re-derive the goldens from the Python side:

```
python3 crates/assay-canonical/tests/reference/build_goldens.py
```

## Claim ceiling

This crate is a canonicalization mechanism plus its conformance fixtures. It makes no claim about
runtime truth, safety, or trust. The set/ordered classification of any given field, and the meaning
of a profile, are per-schema decisions recorded in the contract registry, not asserted here.
